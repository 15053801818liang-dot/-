## 数据流

### 推理查询管线

```
Step 1: 用户输入字符串
          │
Step 2: BoneGuard.check() → 匹配违骨模式则拦截并返回
          │
Step 3: NLMatcher.parse() → 正则模板匹配，输出 Term 列表
          │
Step 4: 如果是学习/命令/控制类（learn_rule, confirm, reject, dream, 等）
          → 直接执行对应操作并返回，不走推理管线。
          │
Step 5: 如果是查询类（direct_query, query, reason_with）
          → 进入推理管线：
            │
            ├── Arbiter.select_methods(goal)
            │     分析目标特征（变量数、谓词名前缀、是否有事实/规则）
            │     返回排序后的候选 ReasoningMethod 列表
            │
            ├── Arbiter.reason(goal, engine)
            │     遍历候选方法执行推理，收集结果
            │     检测冲突 → 按信任度裁决 或 合并结果
            │     记录方法使用历史（供后续反馈学习）
            │
            ├── 若所有方法无结果：
            │     回退到 CoT（最低开销兜底）
            │     检查相似记忆召回
            │
            ├── RealitySupervisor.validate(goal, binding)
            │     事实一致性、逻辑一致性、否定矛盾检测
            │     计算 hallucination_risk
            │
            ├── collect_warnings(trace)
            │     从推理树中提取规则级警告
            │
            └── 返回格式化答案（含验证报告和警告）
```

### 记忆写入管线

```
用户输入 → BoneGuard → NLMatcher
              │
              ▼
      PersistentMemory.log_conversation("user", input)
              │
              ▼
      执行操作（学习/查询/等）
              │
              ▼
      memory.remember(...) 或
      memory.add_skill(...)  （由 AutoSkillLearner 调用）
              │
              ▼
      JSON 文件持久化（IDENTITY / MEMORY / SKILLS / CONVERSATION）
```

### 梦境循环

```
后台线程每 30 秒 → 30% 概率触发 _dream_once()
                          │
                  ├── 一级·回顾：重放最近对话
                  ├── 二级·关联：检查知识库一致性
                  ├── 三级·重构：从事实归纳候选规则
                  │
                  ▼
          存入 pending 队列（不直接修改 KB）
                  │
                  ▼
          用户命令 "梦境确认" / "梦境拒绝" → 生效或丢弃
```

## 组件说明

### 1. SuperBrainAgent（主控）

整合所有子模块，实现感知-决策-执行循环。生命周期：

```
__init__() → 初始化 KB / 4D记忆 / 知识图谱 / 认知引擎 / 梦境引擎 /
              技能学习 / 现实监督 / 仲裁器 / MCP / NLP / 骨骼守护
              加载内置事实和规则
              启动梦境后台线程
               │
run(input) → perceive() → decide() → act()
               │
process_mcp(request) → MCPBridge.process_request()
```

**关键属性**:
- `_pending_context` — 苏格拉底多轮交互的上下文暂存
- `last_result` — 最近一次推理结果的 (goal, binding) 对（供 confirm/reject 使用）
- `last_reasoning` — 最近一次推理的方法和类型记录（供仲裁反馈）

### 2. KB（知识库）

管理事实和规则，提供回溯推理引擎：

| 方法 | 说明 |
|------|------|
| `add_fact(fact)` | 添加原子事实 |
| `add_rule(rule, force)` | 添加规则，自动调用一致性检查 |
| `query_best_with_trace(goal)` | 合一回溯推理，返回最优绑定 + 推理树 |
| `detect_recursive()` | 递归谓词检测（DFS 环检测） |

**回溯算法**:
- 深度优先搜索，支持 `MAX_DEPTH` 和 `MAX_SOLUTIONS` 限制
- 动态重排：先估算子目标匹配数，选择最约束的优先求解
- `specificity_score` 偏好：常量更多的规则得分更高
- 结果按分数排序取最优

**内置谓词** `eq/2`：要求至少一个常量参数，用于等式约束。

### 3. CognitiveEngine（认知引擎）

16 种纯符号推理方法的集合。所有方法共享 `think_log` 列表记录推理过程。

| 方法 | 实现策略 |
|------|----------|
| CoT | 分解目标为子目标链 → 顺序求解 → 合并绑定 |
| ToT | 生成多分支目标变体 → 并行求解 → 选得分最高 |
| ReAct | 5 步迭代：推理 → 行动 → 更新状态 |
| MCTS | 20 次随机模拟 → 分数加权 → 选最优 |
| Socratic | 生成追问列表 → 尝试 KB 求解 → 未解的转入 `_pending_questions` |
| Decomp | 按规则体分解为子目标 → 分批求解 → 合并 |
| Self-Refine | 初始求解 → 一致性验证 → 改进报告 |
| Recursive | 深度优先递归，最多 3 层 |
| Analogy | 相同谓词名的事实类比映射 |
| Abductive | 查找以相同谓词为头的规则 → 反向推导 |
| Inductive | 收集同谓词事实 → 提取公共模式 |
| Dialectic | 解正题 → 推反题 → 取共同点作为合题 |
| Counterfactual | 遍历替代变量绑定 → 假设推理 |
| StepBack | 抽象目标（全变量）→ 求解 → 回退到具体 |
| Contradict | 解正题 → 解反题 → 检测变量绑定冲突 |
| Ensemble | 5 种方法投票 → 选绑定变量数最多的结果 |

### 4. Arbiter（仲裁器）

**职责**: 分析查询目标特征，选择最优推理方法。

**分析维度**:
- 谓词名前缀（`check_*` / `verify_*` → 验证类）
- 变量数量（≥2 或多个参数 → 复杂查询）
- 是否有直接事实或规则匹配
- 谓词名语义（`cause`/`why` → 因果类）

**历史反馈学习**:
- 按 (method, goal_type) 组合追踪成功率
- 滑动窗口 20 次
- 成功率 < 30% 的降权方法仍在候选集中，但调至最后
- 提供 `reset_stats()` 可清空统计

**冲突处理**:
- 多方法结果中，同一变量有不同绑定 → 冲突
- 无冲突 → 合并所有绑定
- 有冲突 → 按置信度（`len(binding) * confidence`）裁决

### 5. DreamEngine（梦境引擎）

**设计原则**: 观察但不修改。所有候选变更进入 pending 队列，用户确认后生效。

| 方法 | 说明 |
|------|------|
| `start()` | 启动后台守护线程（daemon=True） |
| `stop()` | 停止后台线程 |
| `dream_now()` | 立即触发一次完整梦境 |
| `get_pending()` | 查看待确认队列 |
| `apply_pending(i)` | 确认某项（索引从 0 开始） |
| `reject_pending(i)` | 拒绝某项 |
| `compare_pending(i)` | 显示规则 ARG-by-ARG 和 BODY-by-BODY 对比表 |

**候选规则生成**:
- 同谓词事实 ≥ 3 个 → 归纳为泛化规则
- 存在 existing_rule → 添加为替代候选
- 孤儿规则 → 建议删除或补充事实

### 6. MCPBridge（MCP 桥接）

通过 JSON stdin/stdout（`--mcp` 模式）暴露内部能力。

**安全层**:
1. 调用者身份校验（白名单 `trusted_callers`）
2. 骨骼守护文本过滤（所有 params 中的字符串参数）
3. 权限级别（readonly / learn / admin）

**协议格式**:
```json
// 请求
{"method": "query", "params": {"goal": "grandparent(a, _Who)"}, "caller_id": "SanLife"}

// 响应
{"result": "{_Who: c}", "thinking": "...", "success": true}
```

详见 `MCP_API.md`。

### 7. PendingContext（多轮交互上下文）

当苏格拉底式推理需要用户补充信息时，系统暂存推理上下文：

```python
self._pending_context = {
    'goal': goal_term,    # 原始推理目标
}
```

- `perceive()` 检测到 pending → 调用 `feed_answer()`
- 用户输入作为事实加入 KB → 继续推理
- 输入 "cancel" / "取消" 清除 pending
- 不拦截命令（check_consistency, skills, dream 等）

## 设计原则

### 主权先于思考

身份在每次对话启动时声明（`BoneGuard.assert_identity()`），任何试图修改身份的操作都被守卫拦截。

### 骨骼守护持久

基于正则匹配，不依赖 LLM 或外部服务。三层防御：
1. 违词模式匹配（`re.compile`）
2. 硬编码身份白名单（不允许动态添加/删除）
3. 调用日志记录（所有拦截操作记入对话历史）

### 零 API 依赖

全部代码仅使用 Python 标准库（`re`, `json`, `os`, `time`, `hashlib`, `random`, `threading`, `queue`, `copy`, `math`, `uuid`, `sqlite3`, `collections`, `typing`, `dataclasses`, `pathlib`, `enum`, `io`）。

### 记忆为自己

四层记忆参与推理：身份层决定行为边界，陈述层提供持久化事实，技能层存储已验证的模式，对话层用于梦境回顾。所有记忆本地 JSON 持久化，用户完全控制。

### 观察但不修改（梦境引擎）

梦境引擎的候选规则不直接写入知识库，而是进入待确认队列，用户确认后才生效。保证了系统的可审计性和用户主权的最终决策权。

### 自我进化

通过三条学习路径持续成长：
1. **显式学习** — 用户直接教导规则
2. **隐式学习** — 从用户确认/拒绝的推理结果自动归纳/修正
3. **梦境反思** — 后台自动发现模式和问题
4. **自动技能** — 任务执行后评估并创建可复用技能
